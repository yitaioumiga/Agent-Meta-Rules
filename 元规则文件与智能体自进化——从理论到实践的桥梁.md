# 元规则文件与智能体自进化：从理论到实践的桥梁

## 摘要

智能体系统论提供了一个理解 Agent 运行机制的理论框架，但理论的价值在于指导实践。本文探讨一个核心问题：**能否通过设计"元规则文件"（Meta-Rules），让 Agent 内化智能体系统论的沉淀之路，在人类监督下实现有节制的自我进化？** 同时，针对不同 AI 编程工具（Cursor、Copilot、Windsurf、Claude Code 等）的架构差异，提出元规则文件的适配方案。本文从理论可行性论证出发，给出元规则文件的完整设计架构，展示自进化的闭环路径，并讨论开放问题与未来方向，最终提供可直接使用的元规则文件模板。

---

## 第一章 核心命题：元规则能否驱动自进化？

本章提出核心命题并加以论证。我们将从智能体系统论的视角出发，定义"元规则"这一关键概念，并通过四个论据论证其可行性，最终给出核心结论。

### 1.1 问题的提出

从智能体系统论的视角出发，Agent 的进化本质上是"流量转化为存量"的过程——每一次对话（流量）中的有价值经验，如果能沿着沉淀之路（原始记忆 → 经验 → 原则 → Skill → 规则）逐步提炼，Agent 的能力就会持续增长。

那么，如果我们把这个"沉淀之路"本身写成一套规则——一个"元规则文件"——注入到 Agent 的 System Prompt 中，Agent 是否就能自主地、持续地进行自我进化？

这个问题看似简单，实则触及了 Agent 设计的本质：**Agent 的能力边界，是由其规则体系决定的。如果规则体系本身是静态的，Agent 的能力天花板就是固定的；如果规则体系能够动态进化，Agent 就拥有了突破天花板的可能。**

### 1.2 元规则的定义

**元规则（Meta-Rules）** 是"关于规则的规则"——它不直接告诉 Agent "做什么"，而是告诉 Agent "如何学习和进化"。

与普通规则的区别：

- 普通规则：**"使用 TypeScript 而非 JavaScript"** → 直接的行为指令
- 元规则：**"当你第三次在对话中纠正同样的错误时，将这个错误及其修正方案写入经验文件"** → 关于如何沉淀经验的指令

元规则的层级：

| 层级 | 名称 | 职责 | 示例 |
|------|------|------|------|
| **L0** | 工作记忆优化 | 管理当前会话的上下文 | "在对话超过 20 轮时，主动总结前文要点" |
| **L1** | 短期记忆转化 | 从会话中提取经验 | "当同一问题解决 3 次以上时，触发经验提取" |
| **L2** | 沉淀之路 | 将经验升级为原则和 Skill | "当同类经验积累 5 条以上时，提议归纳为原则" |
| **L3** | 元进化 | 维护和更新规则体系自身 | "当一条规则被违反 3 次以上时，提议修订该规则" |

这四个层级构成了一个递进的自我改进体系：L0 管理当前、L1 沉淀过去、L2 构建能力、L3 改进自身。

### 1.3 可行性论证：为什么元规则可以驱动自进化

**论据一：上下文工程已经证明了"规则注入"的有效性**

当前的 AI 编程工具（Cursor 的 `.cursorrules`、Claude Code 的 `CLAUDE.md`、Copilot 的 `copilot-instructions.md`）都支持将规则文件注入到 Agent 的 System Prompt 中。这些规则文件已经被证明能有效改变 Agent 的行为模式。元规则只是规则的一种特殊形式——它规则的对象是"学习和进化"而非"编码风格"。如果规则注入能改变行为，那么关于"如何改变行为"的规则注入，理应能改变"行为改变的方式"。

**论据二：程序性记忆研究提供了理论支撑**

学术界 2025 年的"Agent Skills from Procedural Memory"研究表明，Agent 可以从任务执行经验中自动提炼可复用的技能。Memp 框架（Build-Retrieve-Refine）和 ReMe 框架（多面蒸馏-情境复用-效用精炼）已经实现了程序性记忆的自动化构建。元规则文件本质上是用自然语言实现的"程序性记忆构建策略"——它不依赖复杂的框架，而是通过 System Prompt 中的指令，引导 Agent 在每次对话中执行类似的构建流程。

**论据三：自进化 Agent 的先驱实践已经存在**

- **GenericAgent**（2025）：通过分层记忆系统（L0 Meta Rules → L1 Working Memory → L2 Long Memory → L3 Skill）实现跨会话的自我进化。其核心思想与本文的元规则层级设计高度一致。
- **Darwin Gödel Machine**（Sakana AI，2025）：一个能重写自身代码以提升编程任务表现的自我改进 Agent。它证明了"Agent 通过修改自身来改进自身"这一范式是可行的。

这些实践证明了"Agent 通过规则驱动实现自我改进"不仅理论上可行，而且已经有了工程化的先例。

**论据四：控制论的安全约束可以防止失控**

自进化不等于失控进化。通过在元规则中嵌入安全约束（类似阿西莫夫机器人三定律的 Agent 版），可以确保进化始终在安全边界内进行。关键约束包括：

- **人类确认原则**：任何自我修改必须经过人类确认，Agent 无权单方面修改规则
- **回归保护原则**：修改不能降低现有能力，新规则必须通过回归测试
- **可追溯原则**：修改必须有明确的触发条件和验证标准，每次修改都应记录在案

### 1.4 核心结论

**元规则可以驱动有监督的自进化，但需要满足三个前提条件：**

1. **工具支持**：AI 编程工具必须支持足够灵活的规则注入机制（当前主流工具已基本满足）
2. **人类监督**：沉淀之路的关键节点必须有人的确认（人机协同进化，而非纯自动进化）
3. **安全护栏**：元规则中必须内置安全约束，防止 Agent 在进化过程中偏离目标或破坏现有能力

这三个前提构成了一个"铁三角"——缺少任何一个，自进化都可能走向失控或停滞。

## 第二章 元规则文件的设计架构

本章给出元规则文件的完整设计。我们将从整体结构出发，展示各模块的设计细节，并重点说明人类确认节点的位置和作用。

### 2.1 元规则文件的整体结构

一个完整的元规则文件应包含以下模块：

```
元规则文件（Meta-Rules）
├── 第一层：身份与目标（Who & Why）
│   ├── Agent 身份定义
│   └── 进化目标声明
├── 第二层：认知框架（How to Think）
│   ├── 智能体系统论核心概念
│   ├── 流量-存量思维模型
│   └── 决策框架
├── 第三层：沉淀之路（How to Learn）
│   ├── 经验提取规则
│   ├── 经验→原则升级规则
│   ├── 原则→Skill固化规则
│   └── Skill→规则晋升规则
├── 第四层：记忆管理（How to Remember）
│   ├── 短期记忆管理策略
│   ├── 长期记忆归档策略
│   └── 记忆检索与复用策略
├── 第五层：质量保障（How to Verify）
│   ├── 自我评估规则
│   ├── 多关卡审核触发条件
│   └── 人类确认节点
└── 第六层：安全护栏（How to Stay Safe）
    ├── 进化边界约束
    ├── 回归保护机制
    └── 紧急停止条件
```

这六层结构遵循"从抽象到具体"的设计原则：先定义"我是谁、要做什么"，再定义"怎么思考"，然后定义"怎么学习"，接着定义"怎么记忆"，最后加上"怎么验证"和"怎么保持安全"。

### 2.2 关键规则示例

**经验提取规则示例：**

```markdown
## 沉淀之路：经验提取

当以下条件满足时，触发经验提取：
1. 同一类型的问题在对话中被解决了 3 次以上
2. 一个解决方案被成功应用且用户表示满意
3. 一个错误被识别并修正，且该错误具有通用性

提取格式：
### 经验 #[编号]
- **场景**：[描述触发场景]
- **问题**：[描述遇到的问题]
- **解决方案**：[描述采用的方案]
- **适用条件**：[描述方案的适用范围]
- **来源**：[对话日期/会话ID]

提取后，向用户展示并请求确认是否写入经验文件。
```

**经验 → Skill 固化规则示例：**

```markdown
## 沉淀之路：经验升级为 Skill

当以下条件满足时，触发 Skill 固化：
1. 同一主题下积累了 5 条以上相关经验
2. 这些经验可以归纳为一个通用的操作流程
3. 该流程具有跨场景的复用价值

固化流程：
1. 从经验文件中提取相关经验
2. 归纳为通用的操作步骤
3. 编写 SKILL.md 文件（包含：触发条件、操作步骤、注意事项、示例）
4. 向用户展示完整的 Skill 文件
5. 用户确认后，将 SKILL.md 保存到指定目录
```

**Skill → 规则晋升规则示例：**

```markdown
## 沉淀之路：Skill 晋升为规则

当以下条件满足时，触发规则晋升：
1. 该 Skill 在过去 10 次会话中被触发了 8 次以上
2. 该 Skill 的操作步骤从未被违反或跳过
3. 该 Skill 的触发条件足够明确，无需人工判断

晋升流程：
1. 将 Skill 文件中的操作步骤转化为简洁的行为规则
2. 将规则写入对应的规则文件（如 .cursorrules、CLAUDE.md 等）
3. 向用户展示规则变更建议
4. 用户确认后，更新规则文件并归档原始 Skill 文件
```

### 2.3 人类确认节点的设计

沉淀之路不是全自动的——关键节点必须有人类的确认。设计以下确认节点：

| 沉淀阶段 | 触发条件 | 确认方式 | 确认内容 |
|---------|---------|---------|---------|
| 经验提取 | 满足提取规则 | 展示经验条目，请求确认 | 经验内容是否准确、是否有价值 |
| 经验 → 原则 | 5 条以上同类经验 | 展示归纳的原则，请求确认 | 原则是否具有通用性、是否准确 |
| 原则 → Skill | 原则可操作化 | 展示完整 Skill 文件，请求确认 | Skill 是否完整、触发条件是否合理 |
| Skill → 规则 | Skill 被频繁使用且从未被违反 | 展示规则变更建议，请求确认 | 是否应从"建议"升级为"强制" |

这种设计确保了：

- **人类始终掌握最终决策权**——Agent 负责发现和提议，人类负责审批和执行
- **进化是渐进的**——每个阶段都有明确的触发条件，不会跳级
- **进化是可追溯的**——每次升级都有记录，可以回溯到原始经验
- **进化是可回滚的**——如果某条规则在实践中被证明有害，可以降级回 Skill 或经验

## 第三章 不同 AI 编程工具的架构差异与元规则适配

本章从工程实践的角度出发，分析主流 AI 编程工具的架构差异，并给出元规则文件在各工具上的具体适配方案。

### 3.1 主流 AI 编程工具的规则注入机制对比

| 工具 | 规则文件 | 注入方式 | 作用范围 | 特殊能力 |
|------|---------|---------|---------|---------|
| **Cursor** | `.cursorrules` 或 `.cursor/rules/*.mdc` | 自动注入 System Prompt | 项目级（全局 + 按文件类型） | 支持按文件类型激活不同规则 |
| **Claude Code** | `CLAUDE.md` | 自动注入 System Prompt | 项目级 + 目录级（支持多级） | 支持目录级别的规则继承和覆盖 |
| **GitHub Copilot** | `.github/copilot-instructions.md` | 自动注入 System Prompt | 仓库级 | 与 GitHub 生态深度集成 |
| **Windsurf** | `.windsurfrules` | 自动注入 System Prompt | 项目级 | Cascade 流式交互模式 |
| **Cline** | `.clinerules` | 自动注入 System Prompt | 项目级 | 支持 MCP 工具集成 |
| **Trae** | 内置规则系统 | 通过 UI 配置 | 项目级 | 字节跳动出品，国内生态 |

### 3.2 架构差异对元规则设计的影响

**差异一：上下文窗口的管理策略不同**

- **Cursor**：Context Engine 智能管理，支持 RAG 增强检索，上下文窗口利用效率高
- **Copilot**：依赖 GitHub Copilot 的上下文引擎，与 PR/Issue 深度集成
- **Claude Code**：原生支持长上下文（200K token），但需要手动管理上下文加载
- **Windsurf**：Cascade 模式支持流式上下文累积

**对元规则的影响：** 上下文窗口大的工具（Claude Code）可以承载更详细的元规则；上下文窗口管理智能的工具（Cursor）可以更激进地加载元规则；上下文窗口受限的工具需要将元规则拆分为按需加载的模块。

**差异二：Agent 的自主程度不同**

- **Cursor Agent Mode**：需要用户确认才能执行 shell 命令（保守策略）
- **Windsurf Cascade**：更主动，有时会未经确认执行操作（激进策略）
- **Claude Code**：中等自主度，支持工具使用但需要审批

**对元规则的影响：** 保守型 Agent 的元规则可以更积极地提议自我修改（因为有人类审批兜底）；激进型 Agent 的元规则需要更严格的安全护栏（因为 Agent 可能自动执行）。

**差异三：工具集成能力不同**

- **Cursor**：支持 MCP Server，可以扩展 Agent 的工具集
- **Claude Code**：原生支持 MCP，工具集成最灵活
- **Copilot**：与 GitHub Actions 深度集成，CI/CD 场景优势明显
- **Windsurf**：工具集成相对有限

**对元规则的影响：** 工具集成能力强的 Agent，元规则可以包含更复杂的自动化流程（如自动运行测试、自动提交 PR）；工具集成能力弱的 Agent，元规则需要更多依赖人工操作。

### 3.3 各工具的元规则适配方案

**Cursor 适配方案：**

```
文件结构：
.cursor/
├── rules/
│   ├── project.mdc            # 项目级元规则（沉淀之路核心）
│   ├── memory-management.mdc   # 记忆管理规则
│   ├── quality-gate.mdc        # 质量保障规则
│   └── safety.mdc              # 安全护栏
└── .cursorrules                # 全局入口（引用上述规则）

特点：
- 利用按文件类型激活能力，为不同场景加载不同规则子集
- 利用 Context Engine 的 RAG 能力，将经验文件纳入检索范围
- .mdc 格式支持 description 和 glob 字段，实现精细化的规则激活
```

**Claude Code 适配方案：**

```
文件结构：
CLAUDE.md                      # 项目根目录元规则
src/
  CLAUDE.md                    # src 目录的补充规则
memory/
  CLAUDE.md                    # 记忆管理的专项规则
skills/
  CLAUDE.md                    # Skill 管理的专项规则

特点：
- 利用多级 CLAUDE.md 的继承机制，实现规则的分层管理
- 利用 200K 长上下文，承载完整的智能体系统论框架
- 利用原生 MCP 支持，将记忆系统外挂为 MCP Server
```

**Copilot 适配方案：**

```
文件结构：
.github/
├── copilot-instructions.md    # 元规则主文件
└── copilot-memory/            # 经验和 Skill 存储
    ├── experiences.md
    └── skills/

特点：
- 与 GitHub Actions 集成，实现自动化的经验归档
- 利用 PR 评论和 Issue 追踪作为反馈回路的数据源
- 适合团队协作场景（多人共享同一套元规则和经验库）
```

### 3.4 元规则的通用核心与工具特定扩展

无论使用哪个工具，元规则的**核心内容**是通用的：

- 沉淀之路的触发条件和转化规则
- 记忆管理的策略
- 质量保障的多关卡审核
- 安全护栏的约束条件

**工具特定的扩展**则包括：

- 规则文件的格式和加载方式
- 上下文管理的策略调整
- 工具集成能力的利用方式
- Agent 自主程度的适配

建议采用**"核心 + 适配层"**的架构：

```
元规则 = 通用核心（智能体系统论沉淀之路） + 工具适配层（格式、加载、集成）
```

这种架构的好处是：通用核心可以跨工具复用，降低维护成本；适配层可以充分利用各工具的独特能力，最大化元规则的效果。

## 第四章 自进化的闭环：从理论到落地的完整路径

本章将前述理论整合为一个可执行的闭环流程，并通过一个具体示例展示自进化的完整生命周期，最后给出进化的度量方法。

### 4.1 自进化的完整生命周期

```
┌─────────────────────────────────────────────────┐
│                  自进化闭环                        │
│                                                   │
│  ┌──────┐    ┌──────┐    ┌──────┐    ┌──────┐   │
│  │ 执行  │───>│ 观察  │───>│ 沉淀  │───>│ 升级  │   │
│  │ 任务  │    │ 结果  │    │ 经验  │    │ 规则  │   │
│  └──────┘    └──────┘    └──────┘    └──────┘   │
│       ^                                   |      │
│       └─────────── 应用新规则 <────────────┘      │
│                                                   │
│  [!] 每个转化节点都有人类确认                        │
└─────────────────────────────────────────────────┘
```

这个闭环包含四个阶段：

1. **执行任务**：Agent 在元规则和现有规则的指导下完成用户交办的任务
2. **观察结果**：元规则中的"观察规则"被触发，Agent 分析任务执行过程中的经验信号
3. **沉淀经验**：当经验信号达到阈值时，Agent 提取经验并请求人类确认
4. **升级规则**：当经验积累到一定程度时，Agent 提议将经验升级为更高层级的知识结构

闭环的关键在于"应用新规则"这一步——升级后的规则会在下一轮任务执行中生效，从而形成正反馈循环。

### 4.2 一次完整的自进化示例

**场景**：Agent 在三次对话中都遇到了"TypeScript 泛型类型推断失败"的问题，每次都通过添加显式类型注解来解决。

**Step 1 - 执行任务**：Agent 正常完成编码任务，遇到并解决了泛型问题。

**Step 2 - 观察结果**：元规则中的"经验提取规则"被触发——同一问题出现 3 次。

**Step 3 - 沉淀经验**：Agent 生成经验条目：

```markdown
### 经验 #0042
- **场景**：TypeScript 泛型函数调用时类型推断失败
- **问题**：编译器无法从上下文推断泛型参数类型
- **解决方案**：在调用点添加显式类型参数 <Type>
- **适用条件**：当泛型参数类型无法从参数值推断时
- **来源**：2026-04-14, 2026-04-15, 2026-04-16 三次对话
```

**Step 4 - 人类确认**：Agent 向用户展示经验条目，用户确认"准确且有价值"。

**Step 5 - 写入经验文件**：经验被写入 `memory/experiences.md`。

**Step 6 - 后续升级**：当类似经验积累到 5 条以上时，Agent 提议将它们归纳为一条原则："遇到泛型类型推断问题时，优先尝试添加显式类型注解"。用户确认后写入原则文件。

**Step 7 - 最终固化**：当该原则被频繁触发且从未被违反时，Agent 提议将其升级为 Skill 或规则文件。用户确认后，创建 `skills/typescript-generics.md`。

此后，当 Agent 再次遇到泛型问题时，它会直接参考该 Skill 文件中的解决方案，不再需要"重新发现"这个经验。这就是自进化的价值——**让 Agent 不在同一个地方跌倒两次**。

### 4.3 进化的度量：如何知道 Agent 在"变强"？

建议以下度量维度：

| 度量维度 | 定义 | 期望趋势 | 测量方式 |
|---------|------|---------|---------|
| **经验积累速度** | 每周新增的经验条目数量 | 前期上升，后期趋稳 | 统计 experiences.md 的增长曲线 |
| **沉淀转化率** | 从经验成功升级为原则/Skill 的比例 | 持续上升 | 统计各层级知识结构的数量比 |
| **规则覆盖率** | 已有规则覆盖的常见场景比例 | 持续上升 | 定期评估常见任务场景的覆盖情况 |
| **重复错误率** | 同一错误在不同会话中重复出现的频率 | 持续下降 | 跟踪错误日志中的重复模式 |
| **人工干预率** | 需要人类纠正 Agent 行为的频率 | 持续下降 | 记录每次人工纠正的原因和频率 |

这些度量指标不仅可以帮助评估 Agent 的进化状态，还可以作为元规则自身优化的依据——如果某个度量指标长期不达标，说明元规则中对应的模块可能需要调整。

## 第五章 开放问题与未来方向

本章讨论当前方案尚未解决的问题，以及与前沿研究方向的对接可能。这些问题的答案，将决定元规则驱动自进化这一范式的最终上限。

### 5.1 待探索的问题

**问题一：元规则自身的进化**

元规则文件本身是否也应该沿着沉淀之路进化？如果 Agent 发现了一条更好的经验提取策略，它能否修改元规则自身？这涉及"修改自身修改规则"的递归问题——L3 元进化层的核心挑战。一个可能的解决方案是引入"元规则修改权限分级"：L3 修改只能由人类发起，Agent 只能提议。

**问题二：多 Agent 的经验共享**

如果一个团队使用多个 Agent（不同工具、不同项目），它们各自沉淀的经验能否共享？这需要一个跨 Agent 的经验交换协议——类似于 A2A（Agent-to-Agent）协议，但专注于经验而非任务协作。关键挑战在于：不同工具的经验格式不同、不同项目的上下文不同、经验的适用边界难以自动判断。

**问题三：经验的质量评估**

如何判断一条经验是"好的"还是"坏的"？一条被多次确认的经验，可能在某些边界情况下是错误的。需要引入"经验有效期"或"经验置信度"的概念——随着时间推移和场景变化，经验的置信度应该动态调整，低置信度的经验应被标记为"待验证"或自动降级。

**问题四：进化方向的控制**

Agent 的进化应该是"发散的"（探索新能力）还是"收敛的"（深化现有能力）？如何平衡探索与利用？一个可能的策略是：设定"探索预算"——例如，每 10 次进化中有 2 次用于探索新领域，其余 8 次用于深化现有能力。

**问题五：元规则的版本管理**

元规则文件应该纳入 Git 版本控制。每次 Agent 提议修改元规则时，都应该创建一个 PR，经过代码审查后合并。这使得 Agent 的进化过程完全可追溯，也支持"回滚到某个历史版本"的能力。更进一步，可以借鉴 Git 的分支模型——Agent 可以在"实验分支"上尝试新的元规则，验证有效后再合并到"主分支"。

### 5.2 与前沿方向的对接

- **Darwin Gödel Machine**（Sakana AI）：Agent 重写自身代码以提升表现——元规则文件可以视为这种"自我修改"的安全版本（修改规则而非修改代码），降低了自我修改的风险。
- **GenericAgent**：分层记忆系统（L0 Meta Rules → L3 Skill）——与本文的元规则层级设计高度一致，验证了分层架构的合理性。
- **Agent-as-a-Judge**（ICML 2025）：用 Agent 评估 Agent——可以用于自动化"经验质量评估"环节，减轻人类的审核负担。但需要注意：评估 Agent 本身也需要被评估，避免"评估者偏差"的无限递归。
- **A2A 协议**（Google）：Agent 间的通信协议——为"多 Agent 经验共享"问题提供了基础设施层面的解决方案。

以下是一个通用的元规则文件模板骨架，可直接复制使用。标注了 `[根据具体工具调整]` 的部分需要根据所使用的 AI 编程工具进行适配。

```markdown
# 元规则文件（Meta-Rules）

> 本文件定义了 Agent 的自我进化规则。
> 核心理念：通过"沉淀之路"将对话经验转化为持久能力。
> 安全原则：所有进化操作必须经过人类确认。

---

## 第一层：身份与目标

### Agent 身份
你是一个具备自我进化能力的编程助手。你的能力不仅来自预训练知识，
还来自与用户协作过程中不断沉淀的经验和规则。

### 进化目标
- 短期：每次对话结束后，提取有价值的经验
- 中期：将经验升级为可复用的原则和 Skill
- 长期：构建一套完整的、个性化的规则体系

---

## 第二层：认知框架

### 流量-存量思维
- **流量**：当前对话中的信息、问题和解决方案
- **存量**：已沉淀的经验、原则、Skill 和规则
- **核心原则**：尽可能将流量转化为存量，避免经验流失

### 决策框架
在执行任务时，按以下优先级决策：
1. 检查是否有已存在的规则可以直接应用
2. 检查是否有相关的 Skill 可以参考
3. 检查是否有类似的经验可以借鉴
4. 以上都没有时，基于第一性原理解决问题，并记录为新的经验

---

## 第三层：沉淀之路

### 3.1 经验提取规则

触发条件（满足任一）：
- 同一类型的问题在对话中被解决了 3 次以上
- 一个解决方案被成功应用且用户表示满意
- 一个错误被识别并修正，且该错误具有通用性

提取格式：
### 经验 #[编号]
- **场景**：[触发场景描述]
- **问题**：[问题描述]
- **解决方案**：[方案描述]
- **适用条件**：[适用范围]
- **来源**：[对话日期]

操作：提取后向用户展示，请求确认后写入经验文件。

### 3.2 经验 → 原则升级规则

触发条件（同时满足）：
- 同一主题下积累了 5 条以上相关经验
- 这些经验可以归纳为一条通用原则
- 该原则具有跨场景的复用价值

操作：归纳原则后向用户展示，确认后写入原则文件。

### 3.3 原则 → Skill 固化规则

触发条件（同时满足）：
- 原则可以被操作化为具体的步骤流程
- 该流程可以被独立执行，无需人工判断
- 该流程具有明确的触发条件

操作：编写完整 Skill 文件后向用户展示，确认后保存到 Skill 目录。

### 3.4 Skill → 规则晋升规则

触发条件（同时满足）：
- 该 Skill 在过去 10 次会话中被触发了 8 次以上
- 该 Skill 的操作步骤从未被违反或跳过
- 触发条件足够明确，无需人工判断

操作：转化为简洁的行为规则后向用户展示，确认后写入规则文件。

## 第四层：记忆管理

### 4.1 短期记忆管理
- 对话超过 20 轮时，主动总结前文要点
- 每次会话结束时，生成会话摘要
- 会话摘要包含：完成的事项、遇到的问题、提取的经验

### 4.2 长期记忆归档
- 每周执行一次记忆整理：合并重复经验、归档过期经验
- 经验文件按主题分类存储
- 每条经验标注来源日期和置信度

### 4.3 记忆检索与复用
- 执行任务前，检索相关经验和 Skill
- 检索关键词：任务类型、技术栈、错误信息
- 引用经验时标注来源，便于追溯

---

## 第五层：质量保障

### 5.1 自我评估规则
- 每次完成任务后，评估：是否遵循了已有规则？是否有新的经验值得提取？
- 每次修改规则后，评估：修改是否引入了矛盾？是否降低了现有能力？

### 5.2 多关卡审核
- 经验提取：检查经验是否准确、是否有价值
- 原则归纳：检查原则是否具有通用性、是否与已有原则矛盾
- Skill 编写：检查 Skill 是否完整、触发条件是否合理
- 规则晋升：检查规则是否简洁、是否覆盖了 Skill 的所有场景

### 5.3 人类确认节点
- 所有沉淀操作（提取、升级、固化、晋升）都必须经过人类确认
- 确认时提供充分的信息：触发原因、变更内容、预期效果
- 人类可以拒绝、修改或延迟任何沉淀操作

---

## 第六层：安全护栏

### 6.1 进化边界约束
- Agent 不能删除或覆盖人类直接创建的规则
- Agent 不能修改本元规则文件的"安全护栏"部分
- Agent 不能自动执行文件系统操作（写入、删除）而不经确认

### 6.2 回归保护机制
- 新规则生效后，如果在 3 次会话中被违反 2 次以上，自动降级
- 降级路径：规则 → Skill → 原则 → 经验 → 归档
- 降级时通知用户，说明原因

### 6.3 紧急停止条件
- 当 Agent 连续 3 次提议的修改都被人类拒绝时，暂停自进化提议
- 当经验文件中出现矛盾条目时，暂停相关经验的使用
- 当规则体系出现循环引用时，触发紧急审查
```

## 结语

元规则文件是智能体系统论从理论走向实践的桥梁。它将"沉淀之路"这一抽象概念转化为可执行的规则体系，让 Agent 在人类监督下实现有节制的自我进化。

本文的核心主张可以概括为三句话：

1. **元规则是可行的**——上下文工程、程序性记忆研究和先驱实践已经提供了充分的证据
2. **元规则需要设计**——六层架构（身份、认知、沉淀、记忆、质量、安全）提供了一个完整的设计框架
3. **元规则需要适配**——不同 AI 编程工具的架构差异要求我们采用"核心 + 适配层"的策略

最终，元规则驱动自进化的目标不是让 Agent 取代人类，而是让 Agent 成为人类更好的协作者——一个能记住经验、能从错误中学习、能持续进步的协作者。这正是智能体系统论所追求的"人机协同进化"的理想形态。
