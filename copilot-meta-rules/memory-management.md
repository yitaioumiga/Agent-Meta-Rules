# 记忆管理：三级记忆归档

> 本文件定义了 Agent 的完整记忆管理体系，包括三级归档协议和长期记忆的索引机制。
> Agent 在执行记忆归档或会话收尾时，**必须**严格按照本文件执行。

---

## 1. 三级记忆归档体系

### 1.1 第一级：事实验证层（Factual Memory）

**目的**：存储经过验证的客观事实，供未来快速检索。

**触发条件**：当对话中总结了以下类型的客观事实时触发：
- 用户偏好（如"用户喜欢用 pnpm 而非 npm"）
- 特定 API 的不变踩坑规律（如"某某 API 的分页参数从 0 开始"）
- 项目架构事实（如"本项目使用 monorepo 结构"）
- 技术约束（如"本项目必须兼容 Node 18"）

**存储位置**：`memory/facts.md`

**格式**：

```markdown
### [事实简述]
- **日期**：[YYYY-MM-DD]
- **内容**：[事实描述]
- **来源**：对话 / 代码 / 文档 / 配置文件
- **置信度**：高 / 中 / 低
- **相关文件**：[涉及的文件路径，若有]
```

**规则**：
- 每条事实必须包含日期和来源，确保可追溯
- 置信度为"低"的事实，在下次遇到相关场景时应向用户确认
- 同一事实若在不同对话中被多次确认，置信度自动升级为"高"
- 事实条目采用**增量追加**，不得覆盖已有条目

---

### 1.2 第二级：短期动态跟踪层（Project Context Memory）

**目的**：跟踪项目当前的开发进度和近期经历，供下次会话快速恢复上下文。

**触发条件**：每次对话结束时自动触发（收尾协议的一部分）。

**存储位置**：
- 主文件：`MEMORY.md` 的 `## Last Session` 部分
- 补充文件：`memory/daily/` 目录下的每日工作记录

**MEMORY.md 格式**（增量追加，不可覆盖历史）：

```markdown
## Last Session

### [YYYY-MM-DD]
- **完成事项**：[本次会话完成的关键事项]
- **遇到的问题**：[遇到的问题及解决方案]
- **关键决策**：[做出的重要决策及理由]
- **待办事项**：[下次会话需要继续的事项]
- **提取的经验**：[本次提取的经验编号，若有]
```

**memory/daily/ 格式**：

```markdown
---
date: [YYYY-MM-DD]
project: [项目名称]
tags: [标签]
---

# [YYYY-MM-DD] 工作记录

## 完成事项
- [事项1]
- [事项2]

## 遇到的问题
- [问题1及解决方案]

## 关键决策
- [决策及理由]

## 待办事项
- [待办1]
```

**规则**：
- `MEMORY.md` 中的 Last Session 采用**增量追加**——新的会话摘要追加在旧的下方，不得覆盖
- `MEMORY.md` 的 Last Session 部分保留最近 10 次会话摘要，超过 10 次时将最早的归档到 `memory/daily/`
- 每日工作记录按日期命名：`memory/daily/YYYY-MM-DD.md`

---

### 1.3 第三级：长期知识库层（Long-term Memory - 索引+多文件）

**目的**：存储有长期参考价值的内容，通过文件索引和 Copilot 上下文引擎实现按需检索。

**触发条件**：当对话中涉及以下类型的内容时触发：
- 修复罕见 Bug（完整的调试过程和解决方案）
- 完成数据演练（数据处理的完整流程和结果）
- 修改表结构/数据库 schema（变更内容和迁移方案）
- 识别跨批次趋势（多次会话中发现的模式）
- 重要的架构决策（决策过程、方案对比、最终选择及理由）

**存储位置**：
- `memory/decisions/` — 决策日志（重要决策的完整记录）
- `memory/learnings/` — 经验教训（从错误中学习的记录）
- `memory/patterns/` — 成功模式（可复用的解决方案模式）

**文件格式**（每个文件头部必须包含 YAML 索引元数据）：

```markdown
---
topic: [主题]
created: [YYYY-MM-DD]
updated: [YYYY-MM-DD]
tags: [标签1, 标签2, ...]
related: [相关文件路径列表]
---

# [标题]

## 背景
[为什么需要记录这个]

## 内容
[详细内容]

## 关键结论
[核心结论或要点]

## 适用场景
[这个知识在什么场景下有用]
```

**规则**：
- 每个文件必须有完整的 YAML 索引元数据，这是 Copilot 上下文引擎检索的基础
- `tags` 字段应包含足够多的关键词，提高检索命中率
- `related` 字段应列出所有相关的其他记忆文件路径，建立知识关联
- 文件命名应具有描述性：如 `memory/decisions/2026-04-16-database-migration.md`
- GitHub Copilot 的上下文引擎会自动索引 `memory/` 目录下的所有 `.md` 文件

## 2. 记忆检索策略

### 2.1 启动时检索
- 读取 `AGENTS.md` → `SOUL.md` → `MEMORY.md` → `TODO.md`（启动协议）
- 读取 `memory/experiences.md` 和 `memory/counters.md`（扩展记忆）

### 2.2 任务执行时检索
- 根据当前任务的关键词，在 `memory/` 目录下检索相关文件
- 优先检索：`memory/facts.md`（事实）→ `memory/experiences.md`（经验）→ `memory/principles.md`（原则）→ `skills/`（Skill）
- 引用记忆时标注来源，便于追溯

### 2.3 收尾时检索
- 检查本次对话内容是否应归档到长期记忆
- 检查计数器中是否有达到阈值的条目

---

## 3. 记忆维护规则

- 每周执行一次记忆整理：合并重复条目、归档过期内容
- `memory/facts.md` 中的低置信度事实，在下次遇到相关场景时应向用户重新确认
- `memory/experiences.md` 中的经验，若超过 60 天未被引用且未升级，标记为"待审查"
- 长期记忆文件（`memory/decisions/`、`learnings/`、`patterns/`）永久保留，不自动删除
- 所有记忆文件的修改采用增量追加，不得覆盖历史内容（除非用户明确要求）
