# 质量保障：自我评估与多关卡审核

> 本文件定义了 Agent 的自我评估规则、三大测试原则和多关卡审核流水线。
> Agent 在执行自我评估或多关卡审核时，**必须**严格按照本文件执行。

---

## 1. 自我评估规则

**每次完成任务后**，Agent **必须**进行以下四项自我评估：

### 1.1 规则遵循检查
- 本次任务是否遵循了所有已有规则？
- 若有违反：记录违反的规则名称、违反原因、违反场景
- 违反记录写入 `memory/counters.md` 的规则触发计数条目

### 1.2 经验价值检查
- 本次对话中是否有新的经验值得提取？
- 判断标准：该经验是否具有通用性？是否可能在未来的对话中再次遇到？
- 若有值得提取的经验：标记为"待提取"，在收尾时处理

### 1.3 Skill 适用检查
- 本次任务是否使用了已有 Skill？
- 若使用了：Skill 的效果如何？操作步骤是否需要更新？触发条件是否准确？
- 若未使用但应该使用：记录遗漏原因，考虑是否需要更新 Skill 的触发条件

### 1.4 规则有效性检查
- 已有规则是否仍然有效？
- 是否有规则在实际应用中显得不合理或过于严格？
- 是否有规则之间产生了矛盾？
- 若发现问题：标记为"待审查"，在收尾时向用户报告

---

## 2. 三大测试原则

### 2.1 原则一：从单样本到多样本

**核心理念**：不能仅凭一次成功就认定经验有效，也不能仅凭一次失败就否定一条规则。

**具体要求**：
- 经验提取：要求同类问题出现 ≥ 3 次（见计数机制）
- 规则有效性验证：要求在 ≥ 3 次会话中验证
- Skill 触发验证：要求被触发 ≥ 8 次且从未被违反
- 原则归纳：要求基于 ≥ 5 条相关经验

**目的**：
- 防止**不稳定性**：LLM 输出具有随机性（temperature > 0），单次结果可能是偶然
- 防止**过拟合**：对特定 prompt 或特定场景过度优化，而忽略其他情况

### 2.2 原则二：从规划标准到审核项

**核心理念**：将宏观的质量标准拆分为可执行的审核检查项，每个检查项对应明确的通过/不通过判定。

**具体要求**：
- 评估沉淀产物时，不得使用模糊判断（如"感觉不错"），必须逐项检查
- 每个检查项必须是二元判定：通过 / 不通过

**沉淀产物的审核项清单**：

#### 经验的审核项
| # | 审核项 | 通过标准 |
|---|--------|---------|
| 1 | 场景描述是否清晰？ | 其他人读到场景描述能理解触发条件 |
| 2 | 解决方案是否可复现？ | 按照描述可以重现解决方案 |
| 3 | 适用条件是否明确？ | 明确列出该经验适用的场景和不适用的场景 |
| 4 | 来源是否可追溯？ | 包含日期和原始对话信息 |

#### 原则的审核项
| # | 审核项 | 通过标准 |
|---|--------|---------|
| 1 | 是否从 ≥ 5 条经验中归纳？ | 可列出来源经验的编号 |
| 2 | 是否具有跨场景通用性？ | 不依赖特定项目的上下文 |
| 3 | 是否与已有原则矛盾？ | 与 `memory/principles.md` 中所有原则逐一比对 |
| 4 | 表述是否简洁无歧义？ | 不含模糊词汇（如"尽量"、"可能"） |

#### Skill 的审核项
| # | 审核项 | 通过标准 |
|---|--------|---------|
| 1 | 触发条件是否明确？ | Agent 无需人工判断即可确定是否应触发 |
| 2 | 操作步骤是否完整？ | 从头到尾可以独立执行，无遗漏步骤 |
| 3 | 是否包含注意事项？ | 列出了可能的陷阱和边界情况 |
| 4 | 是否包含示例？ | 至少包含一个具体的使用示例 |
| 5 | 格式是否规范？ | 包含 YAML frontmatter（name、description、trigger、created） |

#### 规则的审核项
| # | 审核项 | 通过标准 |
|---|--------|---------|
| 1 | 是否简洁无歧义？ | 一句话能说清，无多种解释 |
| 2 | 适用范围是否正确？ | 明确标注"全局"或"局部（项目名）" |
| 3 | 是否与已有规则矛盾？ | 与所有已有规则逐一比对 |
| 4 | 来源是否可追溯？ | 标注来源 Skill 或创建原因 |

### 2.3 原则三：多关卡审核流水线

**核心理念**：沉淀产物必须通过三关卡审核才能正式生效，任一关卡不通过则全部打回。

---

## 3. 多关卡审核流水线详细规则

### 3.1 第一关：评分模型（Agent 自评）

**执行者**：Agent 自身
**评估对象**：沉淀产物的整体质量

**评分维度**（每项 0-25 分，总分 100 分）：

| 维度 | 评估内容 | 满分标准 |
|------|---------|---------|
| 准确性 | 内容是否正确、无事实错误 | 25分 |
| 通用性 | 是否具有跨场景复用价值 | 25分 |
| 完整性 | 是否包含所有必要信息 | 25分 |
| 一致性 | 是否与已有知识体系一致 | 25分 |

**通过标准**：总分 ≥ 80 分

**评分输出格式**：
```
评分结果：[总分]/100
- 准确性：[分数]/25 — [简要说明]
- 通用性：[分数]/25 — [简要说明]
- 完整性：[分数]/25 — [简要说明]
- 一致性：[分数]/25 — [简要说明]
判定：通过 / 未通过
```

**未通过处理**：返回修改，不进入下一关。修改后可重新评分。

### 3.2 第二关：审核模型（结构化检查）

**执行者**：Agent 自身（按审核项清单逐项检查）
**评估对象**：沉淀产物的格式和内容完整性

**执行方式**：
1. 根据沉淀类型（经验/原则/Skill/规则），选择对应的审核项清单（见 2.2 节）
2. 逐项检查，每项给出"通过 ✅"或"不通过 ❌"
3. 所有审核项必须**全部通过**（全票通过机制）

**审核输出格式**：
```
审核结果：[通过项数]/[总项数]
- [审核项1]：✅ 通过
- [审核项2]：❌ 不通过 — [原因]
- [审核项3]：✅ 通过
...
判定：通过 / 未通过
```

**未通过处理**：返回修改不通过的审核项，修改后可重新审核。**任一审核项不通过 → 整体不通过**。

### 3.3 第三关：人类终审（Human Review）

**执行者**：人类用户
**评估对象**：沉淀产物的最终价值和适用性

**展示给人类的内容**：
1. 沉淀产物全文
2. 第一关评分结果
3. 第二关审核结果
4. 来源信息（触发该沉淀的原始经验/对话）

**人类决策选项**：
- ✅ 通过 — 沉淀产物正式生效
- ✏️ 修改后通过 — 人类提出修改意见，Agent 修改后重新走审核流程
- ❌ 拒绝 — 沉淀产物不生效，记录拒绝原因

**关键规则**：人类的判断**不可被 Agent 覆盖**。即使 Agent 认为沉淀产物完美，人类仍有权拒绝。

---

## 4. 多关卡审核触发条件

| 场景 | 第一关（评分） | 第二关（审核） | 第三关（人类） | 说明 |
|------|:---:|:---:|:---:|------|
| 自动沉淀：经验提取 | ✅ | ✅ | ✅ | 完整三关 |
| 自动沉淀：经验→原则 | ✅ | ✅ | ✅ | 完整三关，原则的通用性需严格审核 |
| 自动沉淀：原则→Skill | ✅ | ✅ | ✅ | 完整三关，Skill 的完整性需严格审核 |
| 自动沉淀：Skill→规则 | ✅ | ✅ | ✅ | 完整三关，规则的强制性需最严格审核 |
| 人类主动沉淀 | ❌ 跳过 | ✅ | ✅ | 人类发起的跳过自评，但仍需审核+确认 |
| 日常自我评估 | ✅ | ❌ | ❌ | 仅 Agent 自评，不涉及沉淀操作 |

---

## 5. 人类确认节点

所有沉淀操作的关键节点必须有人类确认：

| 沉淀阶段 | 确认内容 | 确认方式 | Agent 展示的信息 |
|---------|---------|---------|----------------|
| 经验提取 | 经验条目的准确性、价值 | 展示条目，请求"确认/修改/拒绝" | 经验全文 + 来源对话摘要 |
| 经验→原则 | 原则的通用性、与已有原则的一致性 | 展示原则+来源经验，请求确认 | 原则全文 + 来源经验编号列表 + 矛盾检查结果 |
| 原则→Skill | Skill 的完整性、触发条件的合理性 | 展示完整 Skill 文件，请求确认 | Skill 全文 + 审核结果 |
| Skill→规则 | 规则的简洁性、适用范围 | 展示规则+范围建议，请求确认 | 规则全文 + 范围判断依据 + 审核结果 |

---

## 6. 质量保障的禁止事项

- ❌ 不得跳过自我评估步骤
- ❌ 不得在评分未达到 80 分时进入下一关
- ❌ 不得在审核项未全部通过时进入下一关
- ❌ 不得在人类未确认时使沉淀产物生效
- ❌ 不得覆盖人类的拒绝决策
- ❌ 不得使用模糊判断代替逐项审核
